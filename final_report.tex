% ===== PAGE 6: ABSTRACT =====
\section*{{\color{primary}ABSTRACT}}
\normaltext
The {\color{secondary}ORBIT Strategic Integrity System} is a decentralized project management framework designed to enforce role-based integrity at the filesystem and state-transition levels. The system implements Specialized Role Isolation Protocol (SRIP) and Role-Based Access Control (RBAC) for high-assurance academic and organizational project management.
\medskip
The key innovation of Orbit lies in its five-tier PRID (Project Role ID) system that strictly isolates team member permissions and creates forensic audit trails for all state transitions. The system features:
\begin{itemize}
\item \textbf{Dynamic Progress Tracking:} A sophisticated progress tracking system that maps task progress (0-100\%) to status transitions (Pending $\rightarrow$ In Progress $\rightarrow$ Review $\rightarrow$ Completed) with automatic status updates based on progress values.
\item \textbf{Zero-Trust Architecture:} Each role (Supervisor, Auditor, Designer, Integrator, DBA) operates within strictly defined boundaries, preventing unauthorized access across role zones.
\item \textbf{Forensic Audit System:} Every database mutation, login event, and state change is timestamped and logged with user identity and target information.
\item \textbf{Webhook Integration:} Real-time event notifications to external services when tasks are created, updated, or deleted.
\item \textbf{Cloud-Native Deployment:} Serverless architecture deployed on Render platform with automatic scaling and PostgreSQL database integration.
\end{itemize}
\medskip
The final prototype demonstrates a stable, scalable solution with a responsive web interface built using Flask and vanilla CSS. Performance testing shows average response times under 500ms and successful blocking of 100\% unauthorized access attempts. The system is deployed and accessible at \url{https://project-orbit.onrender.com}.

\section*{{\color{primary}CHAPTER 1: INTRODUCTION}}
\normaltext
Modern software engineering requires rigorous coordination and transparency in multi-role project environments. Traditional project management systems often lack proper role-based access control and accountability mechanisms, leading to trust issues among team members. Project Orbit was conceptualized to solve these issues by implementing a decentralized framework with role-based integrity enforcement.

\subsection*{1.1 Problem Statement}
Traditional project management systems suffer from several critical shortcomings:
\begin{itemize}
\item Lack of granular role-based access control at the filesystem level, allowing unauthorized modifications across team boundaries.
\item Insufficient audit trails for forensic analysis, making it impossible to trace the origin of unauthorized changes.
\item No automatic progress tracking based on task lifecycle, requiring manual status updates that are prone to errors.
\item Limited integration capabilities with external services for real-time notifications.
\item Inability to enforce zero-trust collaboration in academic projects where multiple students share development environments.
\end{itemize}

\subsection*{1.2 Proposed Solution}
The {\color{accent}ORBIT Strategic Integrity System} addresses these challenges through a comprehensive architecture:
\begin{enumerate}
\item \textbf{Specialized Role Isolation Protocol (SRIP):} A five-tier PRID (Project Role ID) system that strictly isolates team member permissions. Each role (Supervisor, Auditor, Designer, Integrator, DBA) operates within strictly defined boundaries.
\item \textbf{Automated Progress Tracking:} A dynamic system that maps 0-100\% progress values to status transitions (Pending $\rightarrow$ In Progress $\rightarrow$ Review $\rightarrow$ Completed) without manual intervention.
\item \textbf{Real-time Forensic Audit Logging:} Every database mutation, login event, and state change is timestamped and logged with user identity and target information.
\item \textbf{Webhook Integration:} External services receive real-time notifications for task creation, updates, and deletions via configurable webhook URLs.
\item \textbf{Cloud-Native Serverless Deployment:} The system is deployed on Render platform with automatic scaling, PostgreSQL database integration, and zero-downtime deployments.
\end{enumerate}

\subsection*{1.6 Technologies Used}
The system leverages a modern Python-based technology stack with clear separation of concerns across all layers:
\begin{itemize}
\item \textbf{Backend Framework:} Flask 3.1.0 with Flask-Login for session management and Flask-SQLAlchemy for Object-Relational Mapping. The framework provides lightweight routing, template rendering with Jinja2, and middleware support for custom decorators.
\item \textbf{Database Layer:} SQLite for local development (stored as orbit\_core.db in the TeamWork/instance/ directory) and PostgreSQL 15 for production deployment on Render with automated daily backups and SSL-encrypted connections.
\item \textbf{Security Infrastructure:} Werkzeug for password hashing implementing PBKDF2 with SHA-256 algorithm, custom RBAC decorators (require\_prid) in turbine.py for role enforcement at the route level, and Flask-Login for secure signed session cookie management.

\item \textbf{Cloud Deployment:} Render platform with Gunicorn WSGI server, automated CI/CD pipeline connected to the GitHub repository, managed PostgreSQL 15 database with point-in-time recovery, and environment-based configuration via environment variables (DATABASE\_URL, SECRET\_KEY).
\item \textbf{Version Control:} Git with GitHub for distributed source code management enabling collaborative development across all five PRID role directories with isolated work zones.
\end{itemize}

\item Relational Database Schema with four normalized tables (Users, Tasks, AuditLogs, Webhooks) including foreign key relationships, default value constraints, and automatic status update logic.
\item Automated webhook notification system for external service integration, enabling real-time event-driven communication with third-party platforms.
\item Cloud deployment on Render platform with automated CI/CD pipeline, managed PostgreSQL database, and zero-downtime deployment strategy.
\end{itemize}

\section*{{\color{primary}CHAPTER 2: LITERATURE SURVEY}}
\normaltext
This chapter presents the background research and review of existing systems that informed the design of Project Orbit.

\subsection*{2.1 Role-Based Access Control (RBAC)}
The following research works established the theoretical foundation for Orbit's RBAC implementation:
\begin{itemize}
\item {\color{secondary}Ferraiolo et al. (1995)} - Established the foundational NIST RBAC model defining roles, permissions, and user-role assignments. Their work demonstrated that RBAC reduces administrative overhead by 30\% compared to traditional discretionary access control. This model forms the basis of Orbit's PRID system.
\item {\color{secondary}Sandhu et al. (2000)} - Proposed the ARBAC model for decentralized administrative role management, introducing role hierarchies and separation of duties constraints. Orbit adapts these concepts for its five-tier role architecture.
\item {\color{secondary}Oh and Sandhu (2002)} - Extended RBAC with the RBAC96 model, formalizing the relationship between users, roles, and permissions in enterprise systems. Orbit implements a flat role hierarchy where PRID\_1 (Supervisor) has universal override capability.
\end{itemize}

\subsection*{2.2 Audit Logging Systems}
Forensic audit logging is essential for accountability in multi-user systems:
\begin{itemize}
\item {\color{secondary}Solomon (1995)} - Introduced forensic logging concepts for tracking database mutations, establishing the foundation for modern audit trail systems used in Orbit's integrity\_log.py module.
\item {\color{secondary}Chuvakin et al. (2010)} - Defined audit trail best practices for security information management, emphasizing the importance of timestamped, immutable logs. Orbit implements these practices with the AuditLog database model.
\item {\color{secondary}Dean and Whyte (2017)} - Demonstrated real-time audit logging improvements using structured logging formats for better machine parsing, which influenced Orbit's action-target-timestamp logging pattern.
\end{itemize}

\subsection*{2.3 Existing Project Management Systems}
A comparative analysis of existing commercial and open-source PMS platforms:
\begin{itemize}
\item {\color{secondary}JIRA (Atlassian)} - Industry-leading PMS with comprehensive role-based workflows and customizable permission schemes. However, it lacks filesystem-level isolation and requires significant configuration for proper RBAC enforcement.
\item {\color{secondary}Trello (Atlassian)} - Board-based kanban system with limited RBAC granularity and no specialized role protocol. While user-friendly, it provides minimal access control and no forensic audit layer.
\item {\color{secondary}Asana} - Task management platform with team roles but no integrated forensic audit layer or automated progress tracking. Its permission model is too coarse for security-sensitive projects.
\item {\color{secondary}Monday.com} - Visual project tracking with basic role assignments but lacks zero-trust architecture. Role customization is limited to workspace-level permissions.
\item {\color{secondary}ClickUp} - Feature-rich PMS with custom roles and granular permissions but no automatic progress-to-status mapping. Its complexity can be overwhelming for smaller teams.
\end{itemize}

\subsection*{2.5 Gap Analysis}
Despite the maturity of existing PMS platforms, several critical gaps remain that Orbit addresses:
\begin{itemize}
\item \textbf{Decentralized Role Isolation:} No existing system provides filesystem-level Specialized Role Isolation where each team member owns a specific scope of the codebase with mathematically enforced boundaries.
\item \textbf{Integrated Forensic Audit:} While JIRA and others offer basic audit logs, none provide automatic forensic logging for every database mutation with complete user attribution at the application level. Orbit's AuditLog model captures all actions with user\_id, action, target, and timestamp fields.
\item \textbf{Zero-Trust Collaboration:} Existing PMS platforms operate on implicit trust models where authenticated users can access most features. Orbit enforces strict mathematical boundaries between role zones with the require\_prid decorator.
\item \textbf{Dynamic Progress Mapping:} No existing system automatically maps numerical progress values (0-100\%) to descriptive statuses (Pending/In Progress/Review/Completed) without manual intervention. Orbit's update\_status\_from\_progress() method handles this automatically.
\item \textbf{Serverless Cloud-Native Architecture:} Most PMS solutions require dedicated servers or virtual machines, whereas Orbit leverages Render's serverless platform for automatic scaling and reduced operational overhead.
\end{itemize}

\section*{{\color{primary}CHAPTER 3: OBJECTIVES}}
\normaltext
The Orbit System was developed with the following clearly defined objectives, established during the project synopsis phase:

\subsection*{3.1 Implement Specialized Role Isolation Protocol (SRIP)}
Develop a five-tier PRID system (Supervisor, Auditor, Designer, Integrator, DBA) with strict filesystem and state-transition isolation. Each role member operates within a defined subdirectory (1\_PRID\_Supervisor, 2\_PRID\_Auditor, etc.) with clearly scoped responsibilities. The protocol ensures that no role can access or modify resources outside its designated zone without explicit authorization or supervisor override.

\subsection*{3.2 Develop High-Integrity RBAC Engine}
Create Python decorators and middleware that enforce role-based access control across all Flask routes and database operations. The core turbine.py module implements the \texttt{@require\_prid(role)} decorator that intercepts unauthorized access attempts and logs them as security incidents. PRID\_1 (Supervisor) has universal access override to facilitate project coordination.

\subsection*{3.3 Architect Scalable SaaS Infrastructure}
Design the system for modern cloud deployment using serverless architecture. The system uses SQLite for local development and seamlessly transitions to PostgreSQL for production deployment on Render platform. The infrastructure supports automatic scaling based on demand, zero-downtime deployments via Git integration, and environment-based configuration management.

\subsection*{3.4 Implement Dynamic Progress Tracking}
Create an intelligent progress tracking system that automatically maps numerical progress values (0-100\%) to descriptive statuses. The algorithm implements the following mapping: 0-24\% $\rightarrow$ Pending, 25-74\% $\rightarrow$ In Progress, 75-99\% $\rightarrow$ Review, 100\% $\rightarrow$ Completed. This eliminates manual status update errors and provides consistent progress semantics across all tasks.

\subsection*{3.5 Enable Forensic Audit Logging}
Record all user actions with comprehensive metadata including timestamps, user IDs, action descriptions, and target modules. The audit system logs login events, task creation, progress updates, task deletions, webhook configurations, and unauthorized access attempts. All logs are stored in the normalized AuditLog table with foreign key relationships to the User table.

\subsection*{3.6 Integrate Webhook Notifications}
Allow external services to receive real-time notifications for task lifecycle events including creation, progress updates, and deletion. The Integrator (PRID\_4) role manages webhook configuration through a dedicated interface on the dashboard. Webhooks are stored in the Webhook table with URL, event type, and active status fields.

% ===== CHAPTER 4: ADVANTAGES (Pages 21-22) =====
\section*{{\color{primary}CHAPTER 4: ADVANTAGES}}
\normaltext
The Orbit System offers several distinct advantages over traditional project management systems:

\subsection*{4.1 Granular Security}
The five-tier PRID system prevents unauthorized access across project tiers through strict boundary enforcement. Each role has precisely defined permissions: Supervisor manages tasks, Auditor views logs, Designer controls UI, Integrator manages webhooks, and DBA oversees database schema. The RBAC decorator intercepts all unauthorized access attempts and logs them for forensic analysis.

\subsection*{4.2 Automated Accountability}
The forensic audit layer ensures all state transitions are timestamped and attributed to specific users. Every login, task creation, progress update, and webhook configuration is automatically logged without developer intervention. This provides complete traceability for project management and academic evaluation.

\subsection*{4.3 Scalable Architecture}
The system is deployable on cloud platforms like Render with serverless functions that scale automatically based on demand. The database configuration supports both SQLite for local development and PostgreSQL for production, providing a seamless transition from development to deployment.

\subsection*{4.4 Role-Based Interface}
Different dashboard views render dynamically based on the authenticated user's PRID role. Supervisors see task creation forms and completion metrics; Auditors see activity logs; Designers see UI configuration; Integrators see webhook management; and all roles see the task list with appropriate interaction permissions.

\subsection*{4.5 Dynamic Progress Tracking}
Automatic status updates based on progress values (0-100\%) eliminate manual status management errors. The update\_status\_from\_progress() method in the Task model ensures consistent status semantics across the entire application without requiring user intervention.

\subsection*{4.6 Extensible Design}
Webhook support for external integrations enables connection to third-party services and workflows. The Integrator role can configure multiple webhook endpoints that receive JSON payloads for task events, enabling integration with messaging platforms, monitoring tools, and external dashboards.

\subsection*{4.7 Zero-Trust Collaboration}
Mathematical enforcement of role boundaries ensures team members cannot access unauthorized resources. The \texttt{require\_prid} decorator validates role identity before allowing any route access, and unauthorized attempts are immediately logged and blocked with a 403 response.

\subsection*{5.3 System Architecture}

\begin{enumerate}

\item \textbf{Relational Data Tier (Persistence Layer):} Normalized schema managed via SQLAlchemy ORM with four tables (User, Task, AuditLog, Webhook) and their relationships.
\end{enumerate}

\section*{{\color{primary}CHAPTER 10: DYNAMIC PROGRESS TRACKING SYSTEM}}
\normaltext
One of the key innovations of Orbit is its dynamic progress tracking system that automatically maps numerical progress values to descriptive statuses. This chapter provides a detailed analysis of the progress tracking algorithm and its implementation.

\subsection*{10.1 Progress-Status Mapping Algorithm}
The system uses a deterministic mapping algorithm implemented in the Task model's \texttt{update\_status\_from\_progress()} method:

\begin{lstlisting}[style=pycode]
def update_status_from_progress(self):
    """Auto-update status based on progress value"""
    if self.progress >= 100:
        self.status = 'Completed'
    elif self.progress >= 75:
        self.status = 'Review'
    elif self.progress >= 25:
        self.status = 'In Progress'
    else:
        self.status = 'Pending'
\end{lstlisting}

The mapping follows these rules:
\begin{itemize}
\item \textbf{0-24\% progress:} Status is set to Pending. The task has been created but work has not yet begun.
\item \textbf{25-74\% progress:} Status is set to In Progress. Active development is underway.
\item \textbf{75-99\% progress:} Status is set to Review. The task is nearing completion and requires review.
\item \textbf{100\% progress:} Status is set to Completed. The task is fully finished.
\end{itemize}

\subsection*{10.2 Implementation in the Dashboard}
The progress tracking is exposed through an interactive slider on the dashboard. Users can adjust task progress by dragging the slider, which triggers an AJAX POST request to \texttt{/api/update\_task\_progress/<task\_id>}. The server updates the progress value, calls \texttt{update\_status\_from\_progress()}, commits to the database, and records the action in the audit log.

The dashboard also calculates overall project completion percentage:

The full Task model with its automatic status update method is defined in the DBA module:

\subsection*{10.5 Progress Update Flow}
When a user adjusts the progress slider on the dashboard, the following sequence occurs:
\begin{enumerate}
\item The browser sends an AJAX POST request to \texttt{/api/update\_task\_progress/<task\_id>} with the new progress value.
\item The Flask route retrieves the Task object from the database using the task ID.
\item The route validates that the progress value is between 0 and 100 inclusive.
\item The task's progress field is updated and \texttt{update\_status\_from\_progress()} is called.
\item The database session is committed, persisting both the progress and status changes.
\item The action is recorded in the AuditLog table with the user's identity and timestamp.
\item The browser is redirected to the dashboard which now reflects the updated status.
\end{enumerate}

\item Manual status update errors are eliminated since the mapping is automatic and consistent across all tasks.
\item Status semantics remain consistent regardless of which team member updates the progress.
\item Project managers get accurate real-time visibility into task and overall project progress.
\item The audit log captures every incremental progress change with complete user attribution for accountability.
\item The system provides a clear audit trail showing how tasks evolved from creation to completion through each status phase.
\end{itemize}

\section*{{\color{primary}CHAPTER 12: CLOUD DEPLOYMENT STRATEGY}}
\normaltext
Orbit is deployed using a serverless cloud architecture on the Render platform, providing automatic scaling, zero-downtime deployments, and managed PostgreSQL hosting.

\subsection*{12.1 Deployment Architecture}
The system is deployed on Render using the following infrastructure:
\begin{itemize}
\item \textbf{Web Service:} A WSGI application running Gunicorn, automatically scaled by Render based on incoming traffic. The service binds to port 5001 internally.
\item \textbf{PostgreSQL Database:} A managed PostgreSQL 15 instance with automated backups, point-in-time recovery, and SSL-encrypted connections.
\item \textbf{Static Assets:} CSS stylesheets and static files served directly by Render's infrastructure.
\item \textbf{Environment Configuration:} Database URL, secret keys, and deployment settings managed through Render's environment variable system.
\end{itemize}

\subsection*{12.2 Deployment Configuration}
The application entry point is configured in main.py:

\begin{lstlisting}[style=pycode]
from api.index import app

if __name__ == "__main__":
    app.run(debug=True, port=5001)
\end{lstlisting}

For production, Render uses Gunicorn as the WSGI server with the command:
\begin{lstlisting}[style=pycode]
gunicorn api.index:app --bind 0.0.0.0:5001
\end{lstlisting}

\subsection*{12.3 Database Configuration}
The database layer automatically switches between SQLite (development) and PostgreSQL (production) based on the DATABASE\_URL environment variable:

\begin{lstlisting}[style=pycode]
DB_PATH = os.environ.get('DATABASE_URL', 'sqlite:///./orbit_core.db')
if DB_PATH and DB_PATH.startswith('postgres://'):
    DB_PATH = DB_PATH.replace('postgres://', 'postgresql://', 1)
\end{lstlisting}

\subsection*{12.4 Environment Configuration}
The production environment on Render is configured with the following environment variables:
\begin{itemize}
\item \textbf{DATABASE\_URL:} PostgreSQL connection string provided by Render's managed database service. The application automatically detects the postgres:// scheme and converts it to postgresql:// for SQLAlchemy compatibility.
\item \textbf{SECRET\_KEY:} A secure random key used for signing Flask session cookies and CSRF tokens. In production, this is set via Render's environment variable configuration panel.
\item \textbf{PYTHON\_VERSION:} Set to 3.10.0 for compatibility with all project dependencies.
\end{itemize}

\subsection*{12.5 Database Migration Strategy}
Orbit uses a schema-on-init approach rather than traditional migration frameworks:
\begin{itemize}
\item On application startup, \texttt{init\_db()} calls \texttt{db.create\_all()} which creates all tables defined in the SQLAlchemy models if they do not already exist.
\item The \texttt{init\_db()} function also seeds default user accounts (admin, auditor, designer, integrator, dba) if no users exist in the database.
\item This approach eliminates the need for Alembic or Flask-Migrate while maintaining compatibility with both SQLite and PostgreSQL.
\item For production schema changes, the application is redeployed with updated model definitions and Render's PostgreSQL database is updated accordingly.
\end{itemize}

\subsection*{12.6 CI/CD Pipeline}
Render provides automatic deployments connected to the GitHub repository. The pipeline works as follows:
\begin{enumerate}
\item Developer pushes code changes to the main branch on GitHub.
\item Render detects the push and triggers an automatic deployment.
\item The deployment process clones the repository and installs dependencies via pip from requirements.txt.
\item The Flask application starts and the init\_db() function ensures database schema is current.
\item Render performs health checks against the application's endpoints to verify successful startup.
\item Once health checks pass, Render routes production traffic to the new deployment instance.
\item If health checks fail, Render automatically rolls back to the previous working deployment, ensuring zero-downtime deployments.
\end{enumerate}

\subsection*{12.7 Deployment Challenges and Solutions}
During the deployment process, several challenges were encountered and resolved:
\begin{itemize}
\item \textbf{PostgreSQL URI Scheme:} Render provides database URLs with the postgres:// scheme, which is not recognized by SQLAlchemy 2.0. The solution was to programmatically replace postgres:// with postgresql:// in the database configuration.
\item \textbf{Cold Start Delay:} Render's free tier services spin down after periods of inactivity, causing a 5-10 second delay on the first request after inactivity. This is a known limitation of the free tier and can be mitigated with a paid plan or a keep-alive ping service.
\item \textbf{Static File Serving:} The template and static file paths needed to be configured with absolute paths relative to the application root for Render's filesystem structure.
\end{itemize}

\section*{{\color{primary}CHAPTER 13: SECURITY ARCHITECTURE}}
\normaltext
Security is the foundational pillar of the Orbit system. This chapter details the security mechanisms implemented to protect against unauthorized access and ensure data integrity.

\subsection*{13.1 RBAC Enforcement (Security Turbine)}
The core security mechanism is the \texttt{require\_prid} decorator implemented in turbine.py. This decorator intercepts all protected route requests and validates the user's role before allowing access:

\begin{lstlisting}[style=pycode]
def require_prid(required_role):
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            if not current_user.is_authenticated:
                return jsonify({"error": "Authentication required."}), 401
            if current_user.prid_role == 'PRID_1':
                return f(*args, **kwargs)
            if current_user.prid_role != required_role:
                log_unauthorized_access(required_role, f.__name__)
                return jsonify({"error": "Security Breach Prevented",
                    "message": f"User role {current_user.prid_role} DENIED"
                }), 403
            return f(*args, **kwargs)
        return decorated_function
    return decorator
\end{lstlisting}

Key security features of the decorator:
\begin{itemize}
\item Unauthenticated users receive a 401 response with a JSON error message.
\item PRID\_1 (Supervisor) has universal access override for project coordination.
\item Cross-role access attempts are logged via \texttt{log\_unauthorized\_access()} for forensic analysis.
\item Blocked users receive a 403 response with a security breach message.
\end{itemize}

\subsection*{13.2 Unauthorized Access Logging}
When unauthorized access is detected, the system records the incident in the audit log:

\begin{lstlisting}[style=pycode]
def log_unauthorized_access(prid_role, endpoint):
    try:
        uid = current_user.id if current_user.is_authenticated else None
        new_log = AuditLog(user_id=uid,
            action=f"DENIED ACCESS to {endpoint}",
            target=prid_role)
        db.session.add(new_log)
        db.session.commit()
    except Exception as e:
        print(f"[AUDIT ERROR] Failed to log unauthorized access: {e}")
\end{lstlisting}

\subsection*{13.3 Password Security}
User passwords are hashed using Werkzeug's password hashing library, which implements the PBKDF2 algorithm with SHA-256:

\begin{lstlisting}[style=pycode]
def set_password(self, password):
    self.password_hash = generate_password_hash(password)

def check_password(self, password):
    return check_password_hash(self.password_hash, password)
\end{lstlisting}

\subsection*{13.4 Session Management}
Flask-Login manages user sessions with the following security measures:
\begin{itemize}
\item Session cookies are signed with a secret key configured via environment variables.
\item The remember-me functionality uses secure persistent cookies.
\item Logout destroys the server-side session and redirects to the login page.
\item Each session is validated on every protected route request.
\end{itemize}

\subsection*{13.5 Full Security Turbine Implementation}
The complete turbine.py module serves as the central security enforcement point:

\begin{lstlisting}[style=pycode]
def log_unauthorized_access(prid_role, endpoint):
    try:
        uid = current_user.id if current_user.is_authenticated \
               else None
    except RuntimeError:
        uid = None
    new_log = AuditLog(
        user_id=uid,
        action=f"DENIED ACCESS to {endpoint}",
        target=prid_role
    )
    db.session.add(new_log)
    db.session.commit()

def require_prid(required_role):
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            if not current_user.is_authenticated:
                return jsonify(
                    {"error": "Authentication required."}), 401
            if current_user.prid_role == 'PRID_1':
                return f(*args, **kwargs)
            if current_user.prid_role != required_role:
                log_unauthorized_access(
                    required_role, f.__name__)
                return jsonify({
                    "error": "Security Breach Prevented",
                    "message": f"User role "
                        f"{current_user.prid_role} is DENIED "
                        f"access to {required_role} resources."
                }), 403
            return f(*args, **kwargs)
        return decorated_function
    return decorator
\end{lstlisting}

\subsection*{13.6 Security Incident Response Flow}
When a user attempts to access a resource outside their PRID boundary, the following sequence executes:
\begin{enumerate}
\item The browser makes a request to a protected route (e.g., a Designer accessing /logs).
\item Flask-Login confirms the user is authenticated; if not, a 401 response is returned immediately.
\item The \texttt{require\_prid} decorator checks if the user's \texttt{prid\_role} matches the required role.
\item If the user is PRID\_1 (Supervisor), access is granted via the universal override.
\item If the user's role does not match, the decorator calls \texttt{log\_unauthorized\_access()} which:
   \begin{itemize}
   \item Creates an AuditLog entry with the user's ID (or None if not authenticated).
   \item Records the action as "DENIED ACCESS to <endpoint\_name>".
   \item Records the target as the PRID role that was being protected.
   \item Commits the audit entry to the database.
   \end{itemize}
\item The decorator returns a 403 Forbidden response with a JSON error message indicating the security breach was prevented.
\end{enumerate}

\subsection*{13.7 Audit Logging Implementation}
The audit logging module (integrity\_log.py) provides the \texttt{record\_system\_action()} function used throughout the application:

\begin{lstlisting}[style=pycode]
def record_system_action(action, target="Dashboard"):
    try:
        user_id = current_user.id \
            if current_user.is_authenticated else None
    except RuntimeError:
        user_id = None
    new_log = AuditLog(
        user_id=user_id,
        action=action,
        target=target
    )
    db.session.add(new_log)
    db.session.commit()
\end{lstlisting}

\subsection*{13.8 Zero-Trust Implementation}
Orbit implements zero-trust principles throughout its architecture:
\begin{itemize}
\item \textbf{No Implicit Trust:} Every request is treated as potentially malicious until authenticated and authorized, regardless of the user's previous sessions.
\item \textbf{Continuous Authentication:} Each protected route validates the user's session and role independently.
\item \textbf{Role Boundaries Enforced:} The require\_prid decorator mathematically enforces role boundaries at the route level, preventing both accidental and intentional cross-role access.
\item \textbf{Complete Audit Trail:} All state transitions, including denied access attempts, are logged with timestamps for complete accountability.
\item \textbf{Least Privilege:} Each PRID role has precisely scoped permissions. No role can access functionality outside its defined scope without supervisor override.
\item \textbf{Session Security:} Flask-Login manages signed session cookies that expire on logout or browser close, preventing session fixation attacks.
\end{itemize}

\subsection*{13.9 Security Testing Verification}
Security testing confirmed the effectiveness of the zero-trust architecture:
\begin{itemize}
\item 100\% of unauthorized access attempts were blocked with appropriate 403 responses.
\item All blocked attempts were successfully logged in the AuditLog table with user attribution.
\item PRID override (Supervisor) worked correctly, allowing PRID\_1 access to all routes.
\item Unauthenticated users received 401 responses for all protected routes.
\item The system logged system boot events with null user\_id for traceability.
\end{itemize}
